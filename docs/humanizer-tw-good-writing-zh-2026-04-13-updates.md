# humanizer-tw & good-writing-zh v1.1.0 更新說明（2026-04-13）

## 背景

這一版更新源自 `graphify.TW` fork 的繁中在地化實戰：把 `README.zh-CN.md` 用 OpenCC `s2twp` 轉成繁體後，發現**轉換結果仍有大量漏網**，連跑 `humanizer-tw` 和 `good-writing-zh` 也得手動補規則才完整清掃。

實戰暴露出兩個 skill 原本沒覆蓋的情境：

1. **輸入來源是「剛 OpenCC 完的繁中」** —— 病徵分佈跟「AI 直接寫中文」不同。漏網集中在機器字典沒收的台灣用詞（聚類、構建、鏈路、PyPI 包…）、動詞語境誤譯（connect→連線 vs 串連）、半形標點。
2. **技術文件（README、API、CLI）潤飾** —— `good-writing-zh` 完整模式的「錯落」「打破公式化結構」規則會破壞結構化列舉，改出來反而失去技術文件需要的規整性。

本版針對這兩個情境補強。

## humanizer-tw v1.1.0

### 新增 `references/opencc-residuals.md`（111 行）

專門給「OpenCC `s2twp` 剛轉完的繁中」做第二輪清洗的參考檔。涵蓋：

- **OpenCC 漏網詞對照表**（30+ 條）：聚類→分群、構建→建構、鏈路→流程、PyPI 包→套件、後臺→背景、源位置→來源位置、快取項→快取項目、命令（CLI）→指令、註釋→註解、比如→例如⋯
- **動詞語境誤譯**：`connect` 在加入圖/集合語境應為「串連」而非 networking 術語「連線」；`run` 應為「執行」非「運行」；`configure` 應為「設定」非「配置」
- **半形→全形標點規則**：中文句中 `"..."` → 「...」；`,` → ，；`.` → 。；跨語言邊界以主要語言為準
- **錯字表**：「併」（合併）vs「並」（並且）；「象」vs「像」
- **grep 快速檢查指令**：一條 `rg` 把殘留簡中/中國用語 / 錯字 / 半形標點全抓出來
- **實戰範例**：取自 `graphify.TW` README 的前後對照

### 修改 `SKILL.md`

- **類別十：OpenCC 轉換後的漏網**（新章節）點明觸發情境：fork 簡中專案 README、翻譯 GitHub 文件；明確指向 `opencc-residuals.md`，告訴模型「當輸入來源是剛 `opencc` 完或從 zh-CN 翻譯時務必讀這份檔」
- **快速檢查清單**增兩條：OpenCC 漏網詞、中文句中全形標點

## good-writing-zh v1.1.0

### 新增「模式三：技術文件保守模式（conservative）」

說明為什麼需要這個模式：技術文件有大量不該被「潤飾」的內容（程式碼範例、命令列片段、錯誤訊息字串、API 名稱），完整模式的「錯落」「打破公式化」若強行套用，會把結構化列舉改得面目全非，反而傷害資訊傳達。

**只執行四類**：
1. 刪連續三「的」（語意無損時）
2. 刪「進行／加以／實施」（能直接用動詞時）
3. 拆超過 30 字的單句（但見術語連綴例外）
4. 修氣口（標點間連續 15–20 字加逗號斷開）

**停用**：錯落規則、打破公式化段落、段落重組、重寫語義、句尾類型輪換

**絕對不碰**：程式碼區塊、CLI 指令、檔案路徑、URL、Markdown 連結語法、表格結構、英文技術術語、YAML front matter、語言選單、版本標註

### 新增「術語連綴例外」清單

以下情境的長片段視為已有「內部停頓」，即使超過 30 字也不強拆：

- **頓號列表**：`.py .ts .js .go .rs` 每個副檔名就是一個停頓點
- **破折號引介**：「完全多模態 —— graphify 會用 Claude vision…」破折號已分段
- **括號補充**：「做結構分析（類、函式、匯入…）」括號本身是停頓
- **連字符連綴**：「程式碼-論文之間的邊會比程式碼-程式碼邊…」`-` 提供視覺停頓

判斷原則：**去掉中英文/數字/標點後剩不到 15 個中文字**，即使字元計數長也不算違規。

### description 補丁

frontmatter 觸發說明增補「技術文件（README、API 文件、CLI 說明）請使用保守模式」；觸發詞加入「技術文件潤飾」「conservative 模式」。

## 使用建議

`humanizer-tw` 跟 `good-writing-zh` 建議搭配順序不變（先 humanizer → 後 good-writing），但針對兩種情境有不同執行模式：

| 情境 | humanizer-tw | good-writing-zh |
|------|-------------|----------------|
| AI 寫的中文 blog / 社群貼文 | 全面模式（含個性注入） | 完整模式 |
| 簡中專案 fork 的繁中 README | **重點讀 `opencc-residuals.md`** | **保守模式** |
| AI 寫的正式技術文件 | 正式文體模式（停用口語化） | 保守模式 |

## 相關實戰

`graphify.TW` fork 的 `README.zh-TW.md` 就是這兩個 skill v1.1.0 的首個實戰目標。轉換後發現約 14 類殘留需清理（聚類/社區/構建/註釋/比如/後臺/PyPI 包/命令/源位置/快取項/暴露/鏈路/提供方/使用跟蹤），全數歸入 `opencc-residuals.md`。
